# Trigger Design

> Part of `salesforce-apex-quality` — see SKILL.md for the always-on Quick Reference and routing.

- **Business logic in the trigger body** — not unit-testable in isolation, not reusable, and cannot be ordered or toggled. The trigger does event routing only: all contexts route through a single trigger into its handler (`new AccountTriggerHandler().run()`); rules belong in domain/service classes.
- **More than one trigger per object** — execution order across multiple triggers on the same object is undefined, producing intermittent, unreproducible bugs. One trigger per object; managed-package triggers are the only accepted exception.
- **Same-record field updates in the wrong context** — setting fields on the triggering records belongs in a `before` trigger, assigning directly to the `Trigger.new` records with **no DML** (the save that is already in flight persists the change). Doing it in an `after` trigger forces an extra DML, an extra full save cycle, and a recursion guard that would otherwise be unnecessary.
- **Throwing exceptions to block a save** — an uncaught exception in a trigger rolls back the entire batch with a generic fault. Use `record.addError('message')` on the specific records that fail validation: per-record, bulk-safe, and surfaced to the user or `SaveResult` cleanly.
- **No recursion control** — an after-update trigger that updates its own object re-fires until it hits a limit or double-processes records. Guard with a `private static Set<Id>` of processed record Ids, checked and added before processing — not a bare `Boolean`: an `isFirstRun` flag skips the second 200-record chunk of the same bulk operation entirely.
- **Mixed automation on the same object** — Apex triggers, record-triggered Flows, and legacy Workflow Rules on the same object create field-update races and recursive cross-firing with undefined ordering. One automation strategy per object; never silently add a Flow to an object an Apex trigger already owns. If a split is deliberate, follow the official record-triggered automation decision guide pattern — record-triggered Flow as the orchestration layer with high-complexity operations encapsulated in invocable Apex — and document the division of labor explicitly, watching the object's total automation density.
