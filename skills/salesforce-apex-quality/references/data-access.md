# Data Access and Governor Limits

> Part of `salesforce-apex-quality` — see SKILL.md for the always-on Quick Reference and routing.

- **SOQL inside a loop** — each iteration issues a query; at 200 records per trigger batch you cross the 100-query limit (`System.LimitException: Too many SOQL queries: 101`). Query once outside the loop with `IN :idSet`, build a `Map` keyed by the relationship field (e.g., `Map<Id, List<Contact>>` filled from one query over `Trigger.newMap.keySet()`), and read from the map inside the loop. Use a relationship subquery when you need parent and child together.
- **DML inside a loop** — hits the 150-DML-statement limit, and each statement re-fires cascading automation. Accumulate changed records into a `List` and issue one DML after the loop. DML only records that actually changed — compare against `Trigger.oldMap` first. In bulk/batch flows use `Database.update(toUpdate, false)` and process the `SaveResult[]` for per-record errors.
- **Single-record assumption** — Data Loader, Flows, and integrations push in bulk; processing only `records[0]` silently drops the rest. Every public method accepts and processes collections (iterate the whole list); always assume `Trigger.new` holds 200 records. Single-record overloads must delegate to the bulk method, never the reverse.
- **Nested loops instead of a map join** — O(n×m) CPU is the most common cause of `Apex CPU time limit exceeded`, which cannot be raised. Build a `Map` from the inner collection in a single pre-pass, then replace the inner loop with an O(1) map lookup.
- **Non-selective SOQL** — queries without an indexed `WHERE`, without a `LIMIT`, or over-fetching fields (`SELECT FIELDS(ALL)`) degrade as data grows. Filter on indexed fields (`Id`, `Name`, `OwnerId`, lookups, external IDs, custom indexes), select only the fields you need, and add `LIMIT`, `ORDER BY`, and `WITH USER_MODE`. For Custom Metadata Types use `CustomMdt__mdt.getAll().values()` or `getInstance()` — never SOQL.
- **Materializing a huge query result into one `List`** — exceeds the heap limit (6 MB sync / 12 MB async). Use a SOQL `for` loop — `for (List<Account> chunk : [SELECT ...])` — which streams results in 200-record batches via internal `queryMore` calls and keeps the heap flat. This is the documented pattern for very large query results.
- **Callout inside a loop** — each callout counts against the 100-callout-per-transaction limit; 200 records means 200 sequential round trips, then `System.LimitException: Too many callouts: 101`. Aggregate all inputs before the loop into one payload, make one batched callout, and distribute the response back to records in memory by Id. If the external API has no batch endpoint, move the work to a Queueable/Batch chunked within the callout limit. Callouts cannot be made synchronously from a trigger context — see `references/async.md`.
- **Repeated `Schema.describe` calls** — `Schema.getGlobalDescribe()` rebuilds the full metadata map on every call; invoked inside a loop it causes `Apex CPU time limit exceeded`. Cache describe results in a `private static Map` lazily initialized once per transaction.
- **String concatenation with `+=` in a loop** — Apex strings are immutable, so each `+=` allocates a new heap object: O(n²) heap growth and a path to `Apex heap size too large`. Accumulate values into a `List<String>` and call `String.join(parts, separator)` after the loop.

**Collection patterns:** `Map<Id, SObject>` for ID lookups; `Map<Id, List<SObject>>` built in one loop for parent-child grouping; `Set<Id>` for deduplication; relationship subqueries for parent + child in one round trip; `AggregateResult` with `GROUP BY` for rollups.
